## 10. 6-DNA 시장 분류 — 단순 이평선이 아니다
> **"이동평균 골든크로스? 그건 2020년 방식이다"**
>
> 봇이 시장을 읽는 6가지 DNA:
> 1. **허스트 지수** — 추세 지속성 vs 평균회귀
> 2. **섀넌 엔트로피** — 가격 불확실성 (높으면 횡보)
> 3. **청산 압력** — 대규모 청산 발생 여부
> 4. **펀딩 압력** — 롱/숏 편향
> 5. **OI 모멘텀** — 미결제약정 변화 방향
> 6. **청산 밀도** — 전체 거래량 대비 청산 비율
>
> 이 6개를 Z-score 정규화 → 가중합산 → 6가지 시장 상태(강한상승/약한상승/횡보/약한하락/강한하락/변동성) 분류
>
> 에이전트마다 가중치가 다름 (S1은 단기, S4는 스윙)

## 14. 잘 안 된 것들 — 삽질 이야기
> **"실전에서 터진 버그들"**
>
> - **가격 정밀도**: Hyperliquid은 유효숫자 5자리만 허용 → 6자리로 보내면 주문 거부 → `round_price()` 함수 하나 때문에 반나절 디버깅
> - **TP 저장 안 함**: DB에 TP 가격을 저장하면 재시작 시 ATR이 달라져서 엉뚱한 가격에 익절 → "저장하지 않고 매번 재계산"이 정답
> - **캔들 중복 처리**: WS에서 같은 캔들이 여러 번 올 수 있음 → 타임스탬프 변화 감지로 해결 (안 하면 한 캔들에 2번 진입)
> - **맥 sleep**: 노트북 덮으면 봇 프로세스에 SIGTERM → 별도 모니터링 프로세스로 감시
> - **거래소 SL/TP 타입**: "Stop Market", "Take Profit Market" 문자열이 정확히 일치해야 함 → 오타 하나로 손절이 안 걸림
> - **Backfill rate limit**: 46코인 × 4타임프레임 = 184번 API 호출 → 거래소가 차단 → retry + backoff 추가

## 15. 멀티 TP + 트레일링 스탑 전략 (실전 사례 포함)
> **"한 번에 다 파는 게 아니라, 나눠서 판다"**
>
> - TP1 (RR 1.0~1.5배): 포지션 60% 익절
> - TP2 (RR 1.5~2.5배): 나머지 40% 익절
> - TP1 도달 후 → 트레일링 스탑 자동 활성화 (0.8×ATR 간격 추적)
> - 최소 수익 잠금: 진입가 대비 0.3% 이상 확보
>
> **실전 사례 — ASTER SHORT (2/23)**
> ```
> 22:00 진입 @ 0.7045, SL @ 0.72
> 23:02 가격 0.6911 → TP1 도달!
>   → 60% (29개) 즉시 청산 (수익 확보)
>   → SL을 0.72 → 0.69로 이동 (본전 확보)
>   → 트레일링 시작
> 23:06 가격 반등 → 트레일링 SL 터치
>   → 나머지 40% (20개) 자동 청산
>   → 최종 수익: +$0.57
> ```
> SL 0.72까지 안 가고, TP1에서 60% 먹고 트레일링이 나머지를 지켜준 케이스
>
> 포인트: "60%는 안전하게 먹고, 40%는 추세를 탄다. TP1 이후엔 본전 이하로 안 내려감"

## 17. 재시작 자동 복구 (Reconciliation)
> **"봇이 죽었다 살아나도 포지션을 잃지 않는다"**
>
> 재시작 시 자동 수행:
> 1. 거래소에서 실제 포지션 조회
> 2. DB 기록과 대조
> 3. **일치** → 누락된 SL/TP 주문 재배치
> 4. **DB에만 있음** → 거래소에서 이미 청산됨 → DB 정리
> 5. **거래소에만 있음** → 미확인 포지션 → 즉시 청산 (안전)
>
> 지갑 4개 각각 독립 수행
>
> 포인트: "서버 다운, 네트워크 끊김, 재부팅 — 어떤 상황에서도 포지션 관리 연속성 보장"

## 19. 포지션 자동 감지 — SL 체결도 놓치지 않는다
> **"거래소가 손절해도, 봇은 0.5초 안에 안다"**
>
> 문제: SL은 거래소 트리거 주문이라 발동되면 거래소가 알아서 청산. 그런데 봇은 모름 → 텔레그램 알림도 안 오고, DB도 안 맞음
>
> 해결:
> - 30초마다 거래소 포지션 목록 조회
> - 봇이 추적 중인 포지션이 거래소에 없으면 → SL_HIT로 자동 판정
> - DB 기록 + 텔레그램 청산 알림 + 에쿼티 업데이트 즉시 처리
> - 재시작 시에는 Reconciliation이 한 번 더 전수 검사
>
> 포인트: "SL은 거래소가, TP는 봇이 처리. 그리고 봇은 둘 다 빠짐없이 기록한다"

## 20. 동적 RR — 시장 상황에 따라 TP 목표가 달라진다
> **"좋은 셋업에는 공격적으로, 애매하면 보수적으로"**
>
> 6가지 조건을 실시간 점수화 (0.0~1.0):
> | 요소 | 비중 | 예시 |
> |------|------|------|
> | 추세×방향 정렬 | 30% | 강한상승+롱 → 1.0, 역추세 → 0.0 |
> | 시그널 신뢰도 | 15% | MTF 합의도 |
> | MTF 정렬 | 15% | 다중 타임프레임 방향 일치 |
> | 스캔 점수 | 15% | 패턴 강도 |
> | MDD 단계 | 15% | normal → 1.0, survival → 0.0 |
> | 연속 손실 | 10% | 0연패 → 1.0, 4연패+ → 0.2 |
>
> 점수가 높으면 RR 범위 상단(공격적 TP), 낮으면 하단(보수적 TP) 자동 적용
>
> 포인트: "같은 패턴이라도 시장 맥락에 따라 목표가가 다르다 — 이게 적응형 전략"

## 21. 심층 매매 분석 — 봇이 스스로 개선점을 찾는다
> **"일일 리포트에 '왜 졌는지'까지 분석해서 보내준다"**
>
> 매일 09:00 자동 분석 항목:
> - 손절 비율 분석 → SL ATR 배수 조절 제안
> - 보유시간: 수익 거래 vs 손실 거래 평균 비교
> - SL 거리: 수익 때 vs 손실 때 → min_sl_pct 조정 근거
> - Gate 점수별 승률 → 통과 기준 상향/하향 제안
> - 패턴별 손절률 → 약한 패턴 가중치 하향 제안
> - 레짐×방향 콤보 → 역추세 진입 차단 제안
> - TP 도달률 → tp1_rr 조정 제안
>
> 예시 출력:
> ```
> • 손실 거래 SL 거리 0.32% vs 수익 1.50%
>   → 손실 시 SL이 타이트, min_sl_pct 상향 고려
> • T5 POC 손절률 75% (8건)
>   → 해당 패턴 가중치 하향 필요
> • STRONG_DOWNTREND+LONG 승률 0%
>   → 역추세 진입 차단 권장
> ```
>
> 포인트: "단순 승률/PF가 아니라, 어떤 파라미터를 어떻게 바꿔야 하는지 구체적으로 제안한다"

## 30. 데이터 기반 패턴 필터링 — 실전 데이터로 자기 약점을 차단한다
> **"실전 7건 분석해서 약한 패턴을 봇 스스로 걸러낸다"**
>
> 실전 매매 데이터 분석:
> ```
> 총 7건 분석 → 5승 2패 (승률 71%)
> 그런데 2패 모두 T5_POC_MAGNET + SIDEWAYS 조합
>
> T5 POC 패턴이란?
> - 볼륨 프로파일에서 가장 거래 많은 가격(POC)으로 수렴하는 패턴
> - 추세장: POC 터치 후 추세 방향으로 이탈 → 유효
> - 횡보장: POC 주변에서 왔다갔다만 함 → 방향성 없음 → 손절
>
> 조치:
> - A안 (채택): T5_POC + SIDEWAYS만 차단
>   → 손실 2건 제거, 나머지 5건 영향 없음
>   → 예상 PnL +$1.15 개선
>   → 시그널 29% 감소 (불필요한 것만)
>
> - B안 (기각): gate 55점 이상으로 상향
>   → 시그널 71% 감소 → 수익 기회도 날림
> ```
>
> 포인트: "감으로 '이 패턴 안 좋다'가 아니라, 실전 데이터 → 시뮬레이션 → 검증 → 적용. 과학적 자기 개선"
